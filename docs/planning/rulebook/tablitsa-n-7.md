Иными словами, на каждую строку-груз даётся тройка:
- **И-схема** (infrastructure+traction) — применяется ВСЕГДА (и для общего парка, и фактически это инфраструктурная часть);
- **В-схема** (вагонная составляющая) — взимается ТОЛЬКО для вагонов общего парка (РЖД);
- **Номер тарифной схемы 9..13** — схема для собственного (арендованного) спецвагона.

## 3. КАК это входит в расчёт тарифа (plain-Russian)

Это **шаг привязки вагон→схема** (классификатор), который выполняется в самом начале расчёта, ДО подсчёта поясной ставки. Юнита у самой таблицы нет — она лишь выбирает три кода схем. Дальше:

1. **Определить тип спецвагона и груз** → найти строку Таблицы N 7.
2. **Выбрать ветку по собственности вагона:**
   - **Собственный/арендованный:** инфраструктурно-локомотивная часть берётся по **И-схеме** (столбец «И*»); вагонная составляющая (В*) **НЕ платится** (вагон чужой/не РЖД); порожний пробег считается по схеме порожнего рейса (см. `tr1-empty-run.json`, схемы 25/25(1)/26/…). Столбец «Номера тарифных схем» (9..13) — это схема для собственного вагона.
   - **Общий парк (РЖД):** платится И-составляющая (И-схема) **плюс** В-составляющая (В-схема); порожний пробег общего парка уже свёрнут в гружёный тариф.
3. **И-схема** = вход в поясную таблицу инфраструктурно-локомотивной ставки (i-belts: `tr1-i-belts-*.json`, `tr1-i-belts-full.json`). Единица — руб. за отправку как функция (расстояние × минимальная весовая норма схемы).
4. **В-схема** = вход в вагонную поясную таблицу (`tr1-v-belts-full.json`). Единица — руб. за вагон×расстояние.
5. **Коэффициенты из примечаний** (1,28 / 0,9595) применяются как **множитель к итогу по выбранным схемам** — то есть это шаг «прочие коэффициенты», умножается на результат И+В (или И для собственного), а не на одну составляющую. Дословно: «тариф определяется по указанным в настоящей таблице тарифным схемам с коэффициентом …».
6. **Особый порожний для лесовоза общего парка** (строка 7): порожний рейс по схеме **N 25(1)**, но расстояние берётся **60 %** от гружёного расстояния. Это влияет на шаг расчёта порожнего пробега.

## 4. Что РАСШИРЯЕТ или ПРОТИВОРЕЧИТ текущему движку/сидам

### 4.1 РАСШИРЯЕТ (новая детализация, которой нет на диске)

- **Дисамбигуация хоппер-схем.** В `tr1-scheme-classifier-extended.json` все хоппер-коды (ХП/ХМ/ХЗ/ХЦ/ДМ/ОК) свёрнуты в плейсхолдер **`iScheme: "И2..И7", vScheme: "В5..В14"`** — то есть точная привязка НЕ закодирована. Таблица N 7 даёт точные ячейки:
  - цемент (хоппер-цементовоз ХЦ) → **И2 / В5 / 9** (единственная строка с И2!);
  - зерно (ХЗ, хоппер) → **И3 / В8 / 9**;
  - минудобрения (ХМ) → **И3 / В8 / 9** (та же строка 3);
  - агломерат/окатыши, полувагон → **И3 / В11 / 9** (строка 5; для ОК «окатышевоз» сид сейчас ставит И2..И7 — теперь точно В11);
  - думпкар (ДМ, вагон-самосвал) → **И3 / В12 / 9** (строка 9; сид давал В5..В14 — теперь точно В12);
  - щепа (полувагон для техн. щепы) → **И6 / В8 / 12** (строка 12).
  → **Рекомендация:** заменить диапазонные плейсхолдеры в `tr1-scheme-classifier-extended.json` на конкретные (И,В,схема) из этой таблицы для ХЦ/ХЗ/ХМ/ОК/ДМ + добавить строки для бункерных (муковоз В9, нефтебитум В10), автомобилевоза (И5/В8/11), думпкара, щепы.

- **Коэффициент модели 19-9835-01 = 0,9595** — точечный коэффициент на конкретную модель крытого минераловоза. На диске в seed-файлах он не закодирован как множитель привязки (grep по `0,9595`/`0.9595` находит его только в i-belts / classifier-pinned как извлечённый текст, не как правило применения). **Расширение:** добавить в `tr1-special-rules.json` правило model→coeff.

- **Коэффициент 1,28 для 6-осных сочленённых хопперов** (зерно/минудобрения) — аналогично, как множитель привязки к итогу по схемам. Проверить, отражён ли он в `tr1-coefficients.json`/`tr1-special-rules.json` (grep по `1,28` в special-rules ничего не дал — вероятно отсутствует как правило применения).

- **Строка 7 — порожний лесовоз общего парка по N25(1) за 60 % расстояния.** Сид `tr1-empty-run.json` содержит схему 25(1), но 60%-правило именно для лесоматериалов круглых на платформе ОБЩЕГО парка — отдельная оговорка; убедиться, что движок применяет 60 % коэффициент расстояния для этого кейса.

### 4.2 СОГЛАСУЕТСЯ (подтверждает текущую модель)

- Модель «И-часть всегда + В-часть только для общего парка; для собственного вместо В — порожний пробег» — ровно то, что декларирует `_meta.note` в `tr1-scheme-classifier.json` и структура `tr1-scheme-classifier-extended.json` (`vScheme: null` для own). Таблица N 7 это подтверждает дословной шапкой колонок.
- Полувагон/платформа/крытый универсальные → не сюда (Таблица N 6); строки 15–17 Таблицы N 7 — это «прочие спецплатформы» (И3/В1, И3/В2, И3/В14) как fallback.

### 4.3 ПОТЕНЦИАЛЬНОЕ ПРОТИВОРЕЧИЕ (флаг к проверке)

- Сид-плейсхолдер `ОК (окатышевоз) → И2..И7 / В5..В14`. Таблица N 7 кладёт «агломерат и окатыши» в **полувагон** (строка 5: И3/В11), а не в хоппер-диапазон. Если движок резолвит окатышевоз как хоппер с И2, это РАСХОДИТСЯ с источником (должно быть И3/В11). **Требует ручной сверки в `wagon-type.ts` / резолвере.**
- Ни одна строка, кроме строки 1 (цемент-хоппер), не использует **И2**. Если движок назначает И2 другим хопперам — противоречие.
