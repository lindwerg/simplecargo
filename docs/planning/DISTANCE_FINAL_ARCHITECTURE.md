# DISTANCE_FINAL_ARCHITECTURE — финальная слоистая политика разрешения тарифного расстояния

> Статус: **SPEC** (дизайн утверждается этим документом; раздел «Результаты» заполняет Synthesize).
> Дата: 2026-06-10. Ветка: `feat/kniga3-official`.
> Движок: `src/lib/distance/computeDistance.ts` + `src/lib/distance/repository.ts`.
> Контракт: `src/lib/distance/schema.ts` (`DistanceResult`: `km|null`, `legs[]`, `confidence green|yellow|red`, `warnings[]`).

## 0. Принцип (деньги / no-fabrication)

Километр — это деньги. **Уверенно-неправильный km — худший исход**, хуже честного «нет данных».
Поэтому политика слоистая, со строгим приоритетом и строгой моделью уверенности:

| confidence | Что означает | Кто его даёт |
|---|---|---|
| **green** | km подтверждён квитанцией/оплатой оператора (якорь), ИЛИ получен кураторским §2-дисциплинированным графом, который воспроизводит все km-оракулы точно | L1, L2 |
| **yellow** | km вычислен из ОФИЦИАЛЬНО ОПУБЛИКОВАННОЙ ПАРЫ (только прямой lookup, без поиска пути) — данные 2023 г., до Приказа №313; ИЛИ из непроверенной таблицы | L3 |
| **red** | данных нет → числа нет (`km: null`) | L4 |

Ни один слой никогда не «изобретает» km: каждый возвращённый километр трассируется
либо к оплате оператора, либо к опубликованному первоисточнику.

## 1. Слои (в порядке приоритета)

### L1 — проверенные якоря (`special-distances.json` overrides) → GREEN

- Файл: `scripts/seed-data/special-distances.json`, ключ `overrides[]`, форма
  `{a, b, km, note, source}` (ESR-пары, ненаправленные).
- Семантика: пара совпала → **немедленный возврат**, ничего больше не считается.
  Уже реализовано: `computeDistance.ts` шаг 2 («Special ESR-pair override») —
  возвращает `confidence:"green"`, leg `kind:"special"`.
- Текущий состав: 1 якорь — **022207↔050009 = 801** (Красный Сокол → Бологое-Московское),
  выверен оплатой оператора до копейки (16 ваг × 34 906 ₽ × 1.22 НДС = 681 365.12).
  Источник в файле: `operator-payment-verified 2026-06-09; reference-ksb-801.json`. **ОСТАЁТСЯ.**
- Пополнение слоя: только через bulk-импортёр якорей (§4) или вручную с provenance.
  Якорь = оператор подтвердил km оплатой/квитанцией/R-Тарифом. Никаких «прикидок».

### L2 — кураторский §2-дисциплинированный узел-граф (сегодняшний движок) → GREEN

- Это весь существующий путь `computeDistance()`: same-station → shared-узел shortcut →
  перебор (spur-origin × spur-dest) кандидатов с `backboneTerminal` (direct-AS-IS guard +
  anti-undercut floor), фильтры `filterBackBranches` (Layer 1, ручная классификация узлов)
  и `filterGeometricObhodnoy` (Layer 2, геометрический), исключение скоростных линий
  (`tr4-skorostnye-edges.json`), хаб-аддеры +54/+25 c same-radial-line исключением.
- Воспроизводит ground truth ТОЧНО: **2444** (021609→612709), **699** (771500→648503),
  **3108** (023202→528706), **1432** (023202→061108), плюс АЯМ (913403→910000) и
  Крым (856200→856107), плюс все 17 тарифных оракулов.
- Уверенность: `green` при чистом пути; существующие предупреждения сохраняются как есть.
- **`kniga3-official.json` НИКОГДА не вливается в этот граф** (см. §2): загрузчик
  `loadKniga3Official()` в `repository.ts` остаётся ЗА ГЕЙТОМ `DISTANCE_KNIGA3_OFFICIAL`
  (по умолчанию ВЫКЛ) — измерено на реальном движке, что вливание режет три
  квитанция-проверенных оракула (2444→834, 1432→930, 3108→3095).

### L3 — прямой lookup официальной пары (`kniga3-official.json`) → YELLOW

**Только когда L2 вернул red** (маршрут недостижим кураторским графом). L3 — это
fallback ДЛЯ ПОКРЫТИЯ, он **никогда не перекрывает green-результат L2** и никогда
не участвует в выборе минимума.

Механика (строго БЕЗ Дейкстры по официальным рёбрам — это и есть запрещённый сценарий §2):

1. Построить (лениво, отдельно от графа) индекс пар `Map<pairKey(a,b), km>` из
   `scripts/seed-data/kniga3-official.json` (99 127 ненаправленных пар / 13 369 пунктов;
   каждый km — verbatim из CSV первоисточника `scripts/seed-data/kniga3-tp-official.csv`).
2. Прямое совпадение: `(originEsr, destEsr)` есть в индексе ВЕРБАТИМ → `km = pairKm`.
3. Иначе station-leg-adjusted пара: для каждого спура происхождения
   `origin → ТПo (kmO)` и спура назначения `dest → ТПd (kmD)` из `compiled.stationLegs`
   (та же Книга-1 база, что у L2, включая transit-attach/cis-spurs/crimea legs) проверить,
   есть ли пара `(ТПo, ТПd)` ВЕРБАТИМ в индексе. Совпадения есть →
   `km = kmO + pairKm + kmD`, берётся минимум по совпавшим комбинациям
   (минимум по ОПУБЛИКОВАННЫМ парам — это выбор среди verbatim-данных, не поиск пути).
   Вырожденные случаи покрываются автоматически: origin сам ТП → спур 0 км (self-leg).
4. Ничего не совпало вербатим → L3 пасует, проваливаемся в L4 (red).

Результат L3:

- `confidence: "yellow"` (enum уже есть в `schema.ts`, движком пока не использовался).
- `legs`: `spur-origin` (если kmO>0) + `direct` (officialPair ТПo↔ТПd) + `spur-dest` (если kmD>0).
- `warnings`: **«официальная пара 2023 (rlw.gov.ru, до Приказа Минтранса №313 от 12.09.2024) — сверьте по R-Тариф»**.
- Запрещено: транзитивное сцепление пар, любая релаксация, любое «почти совпало»,
  использование пары как ребра графа.

Почему yellow обязателен (проверено по файлам на диске, §3): база официальных пар
датирована 2023-10-12 и НЕ содержит дисциплины обхода скоростных линий — она даёт
Хийтола(022404)↔Бологое(050009) = **499**, тогда как реальный биллинг по якорю = **801**.
Прямая официальная пара может быть легально устаревшей; поэтому она — подсказка
с обязательной сверкой, а не green.

### L4 — red: нет данных → нет числа

Как сегодня: `km: null`, `confidence: "red"`, warning с причиной
(`no kniga1 leg…` / `backbone edge missing…`). Ничего не меняется.

## 2. Точка врезки L3 (где именно в коде)

**Врезка — в `repository.ts::resolveDistance`, НЕ внутри `computeDistance`.**
`computeDistance` остаётся чистой L1+L2-функцией (все 54 distance-теста зовут её напрямую
и не должны видеть L3 без явного опта). Псевдокод:

```ts
// repository.ts
export async function resolveDistance(rawInput: DistanceInput): Promise<DistanceResult> {
  const input = distanceInputSchema.parse(rawInput);
  const data = getData();
  const result = computeDistance(input, data);          // L1 + L2 (как сегодня)
  if (result.km !== null) return result;                // green НИКОГДА не перекрывается
  const fallback = officialPairLookup(input, data);     // L3: прямой lookup, без Дейкстры
  return fallback ?? result;                            // L4: исходный red как был
}
```

Детали:

- `officialPairLookup` живёт в новом модуле `src/lib/distance/officialPair.ts`
  (lookup-логика + типы) и тестируется отдельно; индекс пар лениво кэшируется в
  `repository.ts` рядом с `cachedData` (singleton, как граф).
- Гейт-семантика разводится явно:
  - `DISTANCE_KNIGA3_OFFICIAL=1` — гейт ВЛИВАНИЯ В ГРАФ (`loadKniga3Official`),
    остаётся **ВЫКЛ по умолчанию навсегда** (oracle-regression guard);
  - L3-lookup — отдельный путь, включён по умолчанию (он срабатывает только на
    red-маршрутах и по построению не может сдвинуть ни один оракул), с
    kill-switch `DISTANCE_OFFICIAL_PAIR_FALLBACK=0`.
- Инвариант-тест (обязателен в PR, который это реализует):
  все km-оракулы 2444/699/3108/1432/801 + АЯМ + Крым проходят ЧЕРЕЗ `resolveDistance`
  (не только через `computeDistance`) и остаются green с теми же числами;
  плюс кейс «L2 red → L3 yellow c warning» и кейс «вербатим-пары нет → red».

## 4. Bulk-импортёр якорей — `scripts/import-anchors.mjs` (spec)

Назначение: массовое пополнение L1 проверенными километрами (выгрузки R-Тариф,
квитанции, сверки Анны) без ручного редактирования JSON.

### Вход

TSV или CSV (авто-детект по табу в первой строке), колонки:

```
originEsr,destEsr,km,note
```

- `originEsr`/`destEsr`: 6-значный ESR **или** название станции — название резолвится
  в ESR по `scripts/seed-data/kniga1-sections.json` (точное нормализованное совпадение;
  неоднозначность или промах → строка ОТКЛОНЯЕТСЯ с перечислением кандидатов, ESR
  никогда не угадывается).
- `km`: положительное целое (валидация `Number.isInteger(km) && km > 0`).
- `note`: обязательная человекочитаемая привязка (№ квитанции / скрин R-Тариф / письмо).

### Поведение

1. **Dry-run по умолчанию**; запись только с флагом `--write`. Флаг `--source "<строка>"`
   обязателен при записи — попадает в provenance каждой добавленной записи.
2. Каждая принятая строка добавляется в `scripts/seed-data/special-distances.json` →
   `overrides[]` как `{a, b, km, note, source: "<--source> imported <ISO-дата> by import-anchors"}`.
3. **Dedupe по ненаправленной паре** `(a,b)`:
   - пара уже есть с ТЕМ ЖЕ km → skip (отчёт `dup-same`);
   - пара есть с ДРУГИМ km → **КОНФЛИКТ: строка не пишется**, импорт помечает её в отчёте
     (`conflict: existing 801 vs incoming 799`) — существующий operator-verified якорь
     никогда не перезаписывается молча; разрешение конфликта только вручную;
   - дубль внутри самого входного файла → та же логика.
4. Самопары (`a === b`), битые ESR, km≤0, пустые note → reject с причиной.
5. JSON пишется компактно (стиль файла сохраняется: overrides — массив однострочных
   объектов), `_meta`, `uzel_adders_note`, `routing_rules` не трогаются.
6. Итоговый отчёт: `accepted / dup-same / conflicts / rejected` с построчными причинами;
   exit code ≠ 0 при наличии conflicts (чтобы CI/оператор не пропустил).
7. После записи импортёр напоминает прогнать `npx vitest run src/lib/distance` —
   новый якорь обязан НЕ менять ни один существующий оракул (якоря матчатся раньше
   всего, поэтому конфликт возможен только при пересечении пар — что ловит dedupe).

### Чего импортёр НЕ делает

- Не считает km, не интерполирует, не «подправляет» — только переносит проверенные числа.
- Не пишет в `kniga3-official.json` и не трогает граф.
- Не удаляет существующие записи.

## 5. Сводная таблица решения

```
resolveDistance(origin, dest):
  L1: пара в special-distances.overrides?         → km, GREEN  (якорь, return)
  L2: computeDistance по кураторскому §2-графу    → km, GREEN  (return при km≠null)
  L3: (origin,dest) или (ТПo,ТПd)+споры ВЕРБАТИМ
      в kniga3-official.json?                     → km, YELLOW + warning «пара 2023, до №313 — сверьте R-Тариф»
  L4: иначе                                       → null, RED  (числа нет)
```

Инварианты (нерушимые):

1. L3 никогда не перекрывает не-red результат L2.
2. По официальным парам никогда не выполняется поиск пути (Дейкстра/сцепление).
3. `DISTANCE_KNIGA3_OFFICIAL` (вливание в граф) остаётся ВЫКЛ.
4. Все ground-truth числа неизменны: 2444 / 699 / 3108 / 1432 / 801(якорь) /
   АЯМ 913403→910000 / Крым 856200→856107 + 17 тарифных оракулов.
5. Якорь 022207↔050009=801 остаётся в L1 (operator-payment-verified).
